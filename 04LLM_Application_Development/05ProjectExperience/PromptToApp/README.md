一直在推进一个业务方向：**如何让 Agent 生成并持续向用户实时更新的报告**。

在 2024 年，OpenAI 推出了 Canvas（画布）这一应用形态，LangChain 等框架也陆续开源了相关的画布方案。随后，各大服务商基本都提供了“自由画布”能力，用于承载和展示由 Agent 生成的报告内容。

在我们自身的实现探索中，最初的思路是**面向具体业务场景定制报告**：在容器环境中预置前后端代码模板，再结合用户需求，通过 Prompt / Context Engineering 增强Agent设计能力，并修改源码对 Web 模板进行动态优化，并在容器中启动应用，将对应端口的界面实时展示给用户。

但这种方式最终呈现出来的效果更偏向于一个 **Demo**，距离真正可交付的产品形态还有较大差距。



> 参考：[2025.12.10 让 AI Agent 实时生成个性化交互式可视化体验](https://mp.weixin.qq.com/s/OECH3QXoNayCFxj7M3qDfw)

核心思路是在传统Agent对话流程中，增加**Agent实时生成前端组件**的能力：

- **传统流程**：设计师出稿 → 前端开发组件 → 后端组装数据 → Agent生成数据渲染组件。
- **优化后流程**：设计师将设计稿转化为原型图（或通过AI直接生成HTML原型）→ 基于原型生成Prompt → Agent根据Prompt实时生成前端代码（HTML/WebComponent）并直接渲染。这种方式减少了中间环节，颠覆了传统前端开发模式。

关键实现步骤

1. **设计规范转Prompt** 首先需建立统一的设计规范（如颜色、字体、布局等），并将其转化为AI可读的文本格式（如生成`design-master-prompt.md`）。 例如，通过Prompt引导AI严格遵循设计系统的Token、Tailwind工具类和无障碍规范，生成符合要求的HTML代码。
2. **AI基于规范生成原型图** 通过Slash Command（如`/design-master-prompt`）让AI根据需求生成多个HTML原型版本（例如三栏移动端保险产品领取方式展示组件）。 AI会主动确认模糊需求（如布局类型、设计风格），生成可交互的原型图供业务方选择，减少反复修改。
3. **原型图转组件生成Prompt** 将最终确定的原型图压缩成生产可用的Prompt，确保Agent能稳定生成同类组件。 需注意： 提供Few-Shot示例和反例，避免模型过拟合； 明确组件结构约束（如固定3张卡片、图标使用SVG而非Emoji）； 包含质量检查清单（如布局、内容准确性、无障碍性）。
4. **运行时渲染组件** Agent根据调试好的Prompt，实时将用户数据（如保险条款解读）转换为前端代码，直接渲染出可视化组件。文中展示了通过输入保险领取方式文本，输出交互式卡片的实际效果。
